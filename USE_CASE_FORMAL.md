# USE CASE: Автоматическое разделение расходов поездки

---

## 1. Информация об Use Case

| Атрибут | Значение |
|---------|----------|
| **Название** | Автоматическое разделение расходов поездки |
| **Версия** | 1.0 |
| **Статус** | В разработке |
| **Приоритет** | Высокий |
| **Область применения** | Мобильное приложение |

## 2. Описание

Приложение **полностью автоматически** анализирует банковские транзакции и фотоматериалы участников поездки БЕЗ какого-либо участия пользователя во время поездки. Система определяет релевантные расходы и справедливо их распределяет между участниками.

### Ключевая парадигма: ПОЛНАЯ АВТОМАТИЗАЦИЯ БЕЗ ВЗАИМОДЕЙСТВИЯ

**Во время поездки:**
- ✅ Пользователь НЕ должен ничего выбирать, нажимать или комментировать
- ✅ Приложение работает в фоне, анализируя все расходы
- ✅ Система автоматически понимает, кто на какие расходы участвует
- ❌ Никаких всплывающих окон, запросов или конфирмаций

**После поездки (опционально):**
- 🔧 Пользователь может внести коррективы (если хочет)
- 🔧 Но это не обязательно - базовое распределение уже справедливо
- 🔧 Коррекция занимает 1-2 минуты на всю поездку

### Ключевая особенность: Анализ фотографий чеков и столов

Приложение использует компьютерное зрение для анализа фотографий чеков и столов, чтобы определить:
- **Что было заказано** - распознавание блюд и напитков на фотографиях
- **Кто что съел** - сопоставление фотографий каждого человека с тарелками/бокалами
- **Точное распределение** - вместо просто деления поровну, расчет на основе фактического потребления

Это позволяет достичь справедливого распределения расходов с приемлемой точностью (±5-15%), которой достаточно для друзей. На длительной поездке неточности усредняются.

## 3. Действующие лица (Actors)

### Первичные:
- **Пользователь приложения** - участник поездки, дающий доступ к своим данным

### Вторичные:
- **Банковская система** - источник информации о транзакциях
- **Галерея устройства** - источник фотографий
- **AI/ML движок распознавания лиц** - определение количества людей и их идентификация
- **OCR (Optical Character Recognition) система** - распознавание текста на чеках
- **AI система анализа блюд** - распознавание типов блюд и напитков на фотографиях
- **Computer Vision система** - анализ композиции стола и определение, какое блюдо у какого человека

## 4. Предусловия (Preconditions)

1. ✓ Пользователь установил приложение на мобильном устройстве
2. ✓ Пользователь имеет действующий банковский счет
3. ✓ Пользователь дал разрешение приложению доступ к:
   - Фотогалерее
   - Банковскому аккаунту
   - Датам устройства
4. ✓ Минимум 2 пользователя участвуют в одной поездке

## 5. Основной поток (Main Flow)

### Шаг 1: Инициализация поездки (ОДИН РАЗ)
1. Пользователь открывает приложение
2. Выбирает опцию "Новая поездка"
3. Указывает:
   - Название поездки
   - Дату начала
   - Дату конца
   - Список участников (или добавляет позже)
4. **Нажимает "Начать" - и забывает о приложении на время поездки**

### Шаг 2: ФОНОВАЯ СИНХРОНИЗАЦИЯ (без участия пользователя)

**ВО ВРЕМЯ ПОЕЗДКИ (автоматически в фоне):**
1. Приложение непрерывно подключается к банковскому аккаунту пользователя
2. Автоматически загружает все новые транзакции
3. Сканирует фотогалерею на предмет новых фото
4. Анализирует найденные фотографии (чеки, столы, селфи)
5. **Пользователь НЕ видит ничего, НЕ подтверждает ничего, продолжает отдыхать**

### Шаг 3: АВТОМАТИЧЕСКОЕ КАТЕГОРИЗИРОВАНИЕ РАСХОДОВ (без запросов)

**Приложение самостоятельно определяет тип расхода:**
1. **Фильтрует транзакции по дате поездки**
2. **Идентифицирует предварительные платежи** (бронирование отеля, аренда машины):
   - ✓ Дата платежа предшествует дате поездки
   - ✓ Сумма значительна → "Групповой расход на всех"
3. **Идентифицирует текущие расходы во время поездки:**
   - Рестораны и еда (банк: MCC код 5812, 5814, 5411 и т.д.)
   - Услуги и билеты (подъемники, музеи, экскурсии)
   - Развлечения (парки, клубы, спорт)
   - Личные покупки (аптека, магазин одежды)
4. **Определяет ТИП расхода автоматически - пользователь НЕ выбирает**

### Шаг 4: Анализ фотоматериала
1. Приложение анализирует фотографии, сделанные в дни поездки
2. Использует AI/ML для распознавания:
   - Количества людей на фото
   - Личности участников (если доступны)
   - Контекста (аэропорт, ресторан, активность и т.д.)
3. Создает "временную шкалу присутствия" для каждого участника

### Шаг 4.1: Анализ фотографий чеков (NEW)
1. Приложение определяет фотографии чеков в галерее
2. Использует OCR для распознавания текста:
   - Название ресторана/магазина
   - Дата и время
   - Список позиций и их стоимости
   - Общую сумму
3. Сопоставляет информацию с банковской транзакцией
4. Создает детальный реестр позиций по каждому платежу

### Шаг 4.2: Анализ фотографий стола и блюд (NEW)
1. Приложение определяет фотографии со столом и едой
2. Использует Computer Vision для распознавания:
   - Количества блюд/тарелок/напитков
   - Типа каждого блюда (пицца, паста, суп и т.д.)
   - Позиции блюд относительно людей на фото
3. Создает "карту стола" - какое блюдо перед каким человеком
4. Получает примерную стоимость каждого блюда из меню ресторана (если доступно)

### Шаг 4.3: Соотнесение блюд с участниками (NEW)
1. Приложение анализирует фотографии каждого участника
2. Использует CV для определения:
   - Какие блюда человек держит или перед ним на столе
   - Какие напитки он потребил
   - Примерный размер порции (полная, половина и т.д.)
3. Создает "профиль потребления" для каждого участника
4. Если человек не на фото со столом, использует косвенные данные (время в ресторане, присутствие на других фото)

### Шаг 5: Интеллектуальное распределение расходов

#### 5.1 Для расходов с анализом чеков и фотографий блюд:
1. Приложение использует "профили потребления" каждого участника
2. Распределяет стоимость блюда пропорционально потреблению:
   - Если 5 человек заказали 5 пицц и на фото видно, кто что съел → точное распределение
   - Если заказали 1 пиццу на 5 человек → каждый получает 1/5
   - Если один человек съел 2/3 пиццы, другой 1/3 → расход распределяется пропорционально
3. Для напитков:
   - Анализирует, кто какие напитки на себе видит
   - Если человек заказал дорогое вино один → платит сам
   - Если дешевая вода на столе → распределяется поровну между присутствующими
4. Итоговая стоимость рассчитывается как сумма справедливых долей

#### 5.2 Для расходов без анализа (без фото чека/стола):
1. Приложение определяет время каждой транзакции
2. Сопоставляет транзакции с фотографиями присутствия:
   - Если на фото в близкий к транзакции момент времени присутствуют люди X, Y, Z
   - Расход распределяется между ними поровну
3. Исключает "личные" расходы:
   - Если человек купил что-то один (вода в аэропорту)
   - И на фото в тот же момент он один
   - Это не добавляется в общую кассу

### Шаг 6: ФОНОВЫЙ РАСЧЕТ (во время поездки)
1. **Приложение непрерывно пересчитывает окончательные платежи:**
   - Анализирует новые расходы по мере их появления
   - Применяет автоматические правила
   - Обновляет окончательный расчет
2. **Пользователь НИЧЕГО НЕ ВИДИТ И НЕ ПОДТВЕРЖДАЕТ**
   - Система просто работает в фоне
   - Никаких уведомлений о подтверждении

### Шаг 7: ПОКАЗ РЕЗУЛЬТАТОВ (ТОЛЬКО ПОСЛЕ ПОЕЗДКИ)
1. **Поездка закончилась → пользователь открывает приложение**
2. **Видит окончательные платежи:**
   - Детальный список всех расходов с категориями
   - Как каждый расход был распределен и почему
   - Итоговую сумму платежей
3. **Может внести коррективы (опционально):**
   - Редактировать категоризацию расхода
   - Изменить кого включать/исключать
   - Добавить комментарий
4. **Нажимает "Завершить" и все - готово!**

## 6. Альтернативные потоки (Alternative Flows)

**⚠️ ВАЖНО:** Все эти потоки доступны ТОЛЬКО ПОСЛЕ завершения поездки! Во время поездки никаких коррекций и выбора.

### 6.1 Ручное добавление расходов (POST-TRIP)
**Условие**: Пользователь хочет добавить расход, который не был автоматически обнаружен

**Поток**:
1. Пользователь выбирает "Добавить расход вручную"
2. Указывает сумму, категорию, дату
3. Выбирает участников, включенных в расход
4. Приложение добавляет расход в расчет

### 6.2 Коррекция автоматического распределения (POST-TRIP)
**Условие**: Приложение неправильно определило участников расхода

**Поток**:
1. Пользователь нажимает на расход
2. Видит предложенное распределение
3. Может отредактировать список участников
4. Подтверждает изменения

### 6.3 Исключение расхода (POST-TRIP)
**Условие**: Расход был автоматически добавлен, но не должен быть разделен

**Поток**:
1. Пользователь нажимает на расход в отчете после поездки
2. Выбирает "Исключить из общих расходов" или "Только для меня"
3. Приложение пересчитывает окончательные платежи

### 6.4 Добавление участников позже (POST-TRIP)
**Условие**: В начале поездки были не все участники (ошибка в настройке)

**Поток**:
1. Пользователь нажимает "Добавить участника" в отчете после поездки
2. Выбирает контакт и указывает дату присоединения
3. Приложение пересчитывает расходы, исключая те, что произошли до присоединения

### 6.5 Корректировка при наличии фотографий чеков (POST-TRIP)
**Условие**: Система неправильно распознала блюда на фото или распределила их неверно

**Поток**:
1. Пользователь нажимает на расход ресторана
2. Видит:
   - Фотографию чека
   - Распознанный список позиций и их стоимость
   - "Карту стола" - кто что съел
   - Рассчитанное распределение
3. Может вручную отредактировать:
   - Какое блюдо на самом деле кому принадлежит
   - Кто участвовал в общих расходах (напитки, закуски)
   - Исключить личные заказы из группового расчета
4. Приложение автоматически пересчитывает справедливое распределение

### 6.6 Загрузка фотографий чека/стола вручную (POST-TRIP)
**Условие**: Пользователь хочет добавить информацию о расходе через фотографии

**Поток**:
1. Пользователь нажимает на расход
2. Выбирает "Загрузить фото чека" или "Загрузить фото стола"
3. Делает/выбирает фотографию
4. Приложение анализирует фото и автоматически заполняет информацию
5. Пользователь может утвердить или отредактировать результаты

## 7. Постусловия (Postconditions)

### При успешном выполнении:
1. ✓ Все релевантные расходы автоматически идентифицированы
2. ✓ Каждый расход автоматически распределен между участниками по справедливым правилам
3. ✓ Пользователь НИЧЕГО НЕ ДЕЛАЛ во время поездки (полная автоматизация)
4. ✓ После поездки пользователь видит:
   - Сколько каждый потратил
   - Сколько каждый должен/ему должны
   - Детальное объяснение каждого расхода
5. ✓ Окончательная таблица платежей готова и справедлива (±5-15% точность)
6. ✓ Пользователь может внести коррекции если хочет (опционально)
7. ✓ Результаты могут быть поделены между участниками

## 11. Пример использования (Scenario)

### Сценарий: Поездка в Барселону

**Участники**: Анна, Боб, Карл, Диана (4 человека)

**Дата**: 1-7 июня

**Расходы**:
1. **31 мая** - Бронирование квартиры: €600 (Анна)
   - Разделено между 4 участниками: €150 каждому
   
2. **1 июня** - Аренда машины: €400 (Боб)
   - На фото все 4 человека → €100 каждому

3. **2 июня** - Ресторан: €120 (Карл)
   - На фото: Карл, Диана, Анна → €40 каждому
   - Боб не на фото → €0

4. **3 июня** - Подъемник: €80 (Анна)
   - На фото: Анна и Боб → €40 каждому
   - Карл и Диана не ходили → €0

5. **3 июня** - Вода в магазине: €5 (Диана)
   - На фото только Диана одна → только Диана платит

**Итого расходов по человеку**:
- Анна потратила: €600 + €120 = €720
- Боб потратила: €400 + €80 = €480
- Карл потратил: €120
- Диана потратила: €5

**Справедливое распределение общих расходов** (€595 на 4):
- На каждого по: €148.75

**Результат**:
- Анна должна получить: €720 - €148.75 × 3 = €271.75
- Боб должен получить: €480 - €148.75 × 3 = €73.75
- Карл должен заплатить: €148.75 × 3 - €120 = €325.25
- Диана должна заплатить: €148.75 × 3 - €5 = €441.25

**Окончательные платежи**:
- Карл → Анна: €271.75
- Диана → Анна: €73.75 + €41.25 = €115 (остаток)
- Диана → Боб: €326.25

## 11.2 Сценарий со анализом фотографий чеков и столов (NEW)

**Участники**: Влад, Инна, Таня (3 человека)

- Время: 20:30
- Сумма: €87
- Место: "Trattoria Roma"
- Участники (из фото в ресторане): все трое видны на фото → возможно €29 на человека

**Шаг 2: Загрузка фотографии чека (NEW)**
- Влад загружает фото чека
- OCR система распознает:
  - Паста Карбонара (€18)
  - Паста Болоньезе (€16)
  - Паста Примавера (€14)
  - Салат Цезарь (€12)
  - Вода 1л (€4)

- Загружается фото со столом
- Computer Vision система определяет:
  - Перед Владом: чаша с пастой (похожа на Болоньезе)
  - Перед Инной: чаша с пастой (похожа на Карбонара)
  - Перед Таней: чаша с пастой (похожа на Примавера)

**Шаг 4: Анализ, кто что ел (NEW)**
- Система анализирует фото каждого участника индивидуально:

## 11.3 Примечание о типических кейсах для анализа фотографий

### Когда фотографии особенно полезны:

1. **Разные блюда и цены**
   - Кто-то берет дешевую куриную котлету (€8), кто-то морского окуня (€22)
   - Справедливое распределение вместо простого деления пополам

2. **Разные напитки**
   - Один заказал воду (€2), другой премиум вино (€18)
   - Без фото оба платят по €10, с фото каждый платит по справедливости

3. **Спорные ситуации**
   - На фото видно, что один человек заказал два блюда
   - Если бы это было спорно → фото разрешает конфликт

4. **Люди, которые приходили позже**
   - На групповом фото видно, что четвертого человека нет на первом фото стола
   - Система исключает его из первого раунда расходов

5. **Алкогольные и безалкогольные напитки**
   - На фото видно, кто пил алкоголь, кто воду
   - Справедливое разделение счета за напитки

## 13.1 КРИТИЧЕСКАЯ ФИЛОСОФИЯ: Полная Автоматизация Без Взаимодействия

### Основная парадигма:

**Приложение должно работать так, как если бы его вообще не было во время поездки.**

#### Что пользователь НЕ ДОЛЖЕН делать во время поездки:

❌ **НИКОГДА** спрашивать "кто это был?"
❌ **НИКОГДА** спрашивать "это расход для группы?"
❌ **НИКОГДА** показывать уведомления "подтвердите расход"
❌ **НИКОГДА** требовать выбора категории расхода
❌ **НИКОГДА** просить выбрать участников
❌ **НИКОГДА** показывать диалоги или всплывающие окна
❌ **НИКОГДА** запрашивать описание или комментарий

**ПОЛНАЯ ТИШИНА И ФОНОВАЯ РАБОТА**

#### Почему это важно:

- Люди на поездке хотят отдыхать, а не заполнять формы
- Чем меньше интерактивности → тем выше вероятность использования
- Точность ±5-15% автоматически > 0% если пользователь забудет что-то отметить
- Друзья готовы к небольшим неточностям, если общее справедливо

#### Когда пользователь может видеть информацию:

✅ ТОЛЬКО ПОСЛЕ ПОЕЗДКИ:
- Пользователь открывает приложение после возвращения
- Видит детальный отчет всех расходов
- Может сделать коррекции если хочет (но это опционально)
- На коррекции уходит 1-2 минуты максимум

#### Оценка точности vs Автоматизации:

| Сценарий | Точность | Автоматизм | User Experience | Выбор |
|---------|----------|-----------|-----------------|-------|
| Полностью ручной ввод | 100% | 0% | ❌ Ужасно, люди забывают | ❌ |
| Ручной с подсказками | 95% | 20% | ⚠️ Постоянные вопросы | ❌ |
| Полностью автоматический | 85-90% | 100% | ✅ Идеально, ноль отвлечений | ✅✅✅ |
| **НАШЕ ПРИЛОЖЕНИЕ** | **±5-15%** | **100%** | **✅ Идеально для друзей** | **✅ ВЫБОР** |

### Примеры работы "без взаимодействия":

#### Пример 1: Ресторан
1. Одного человека банк списал €50 в ресторане
2. На фото в это время 4 человека вместе на столе
3. **Система БЕЗ ВОПРОСОВ распределяет €50 на 4** 
4. Пользователь ничего не делает, ничего не подтверждает

#### Пример 2: Дорогое вино
1. Банк списал €80 за вино в ресторане
2. На фото видно, что только один человек пьет это вино
3. **Система БЕЗ ВОПРОСОВ назначает €80 только ему**
4. Никаких диалогов, никаких подтверждений

#### Пример 3: Отель
1. Один человек забронировал отель за €200 на неделю
2. Система видит это по дате (до поездки) и сумме (значительная)
3. **Система БЕЗ ВОПРОСОВ распределяет €200 поровну между всеми на 7 дней**
4. Полная тишина в приложении

## 14. Разница между "справедливо" и "идеально"

### Технические требования:

**Для базовой функциональности:**
- Приложение должно работать без постоянного интернета (синхронизация по требованию)
- Необходима поддержка нескольких валют для международных поездок
- Нужна возможность совместного доступа к отчетам между участниками
- Планировать интеграцию с популярными платежными системами

**Для анализа фотографий (NEW - КРИТИЧНО):**
- **OCR движок**: Google Cloud Vision API, Azure Computer Vision или локальная обработка с Tesseract
- **Computer Vision**: TensorFlow Object Detection для определения блюд и людей
- **Face Recognition**: OpenFace или similar для идентификации участников
- **Image Classification**: Training на наборах данных рестораторов для распознавания блюд
  - Категории: основные блюда, напитки, салаты, закуски, десерты
  - Разные кухни: итальянская, азиатская, мексиканская и т.д.
- **Временная синхронизация**: Использование EXIF данных фотографий для точного соответствия с транзакциями

### Стратегия внедрения анализа фотографий:

**Фаза 1 (MVP)**: 
- Базовое распознавание чека (OCR)
- Простое распознавание людей на фото
- Ручное назначение блюд участникам

**Фаза 2 (v1.1)**:
- Автоматическое распознавание типов блюд
- AI-поддержанное сопоставление блюд с людьми
- Поддержка нескольких языков в OCR

**Фаза 3 (v1.2)**:
- Обучение модели на меню популярных ресторанов
- Голосовое объяснение распределения ("Влад съел пасту (€18), Инна вино (€8)...")
- Поддержка нескольких чеков в одном заказе

### Прием данных от пользователя:

**Два потока загрузки:**

1. **Активный (с UI)**:
   - Пользователь нажимает "Добавить чек" или "Добавить фото стола"
   - Делает фото или выбирает из галереи
   - Приложение обрабатывает и показывает результат

2. **Пассивный (фоновый)**:
   - Приложение автоматически сканирует новые фотографии в галереe
   - Если обнаружены чеки, автоматически обрабатывает
   - Уведомляет пользователя о найденных расходах

### Качество vs Справедливость: Философия Полной Автоматизации

**КЛЮЧЕВОЙ ПРИНЦИП:** Система должна быть максимально справедлива БЕЗ участия пользователя.

Система должна достичь **85-90% справедливости**, а не 100%:
- Плюс-минус погрешность в ±5-15% полностью приемлема для друзей
- На длительной поездке (неделя) неточности усредняются и становятся незаметны
- ✅ Лучше "приблизительно справедливо" автоматически
- ❌ чем "идеально справедливо вручную" (которое никто не будет делать)
- **Пользователь МОЖЕТ отредактировать после поездки, но это опционально**

**Истина для друзей:**
- €250 на человека ± €25 = все довольны ✅
- €250 на человека ± €0 но 5 часов расчетов = никто не будет делать ❌

**На практике:**
- Во время поездки: **ПОЛНАЯ АВТОМАТИЗАЦИЯ**, 0 взаимодействия пользователя
- После поездки: Показать результат, пользователь может корректировать (опционально, < 2 минут)

### Поддержка различных сценариев:

✓ **Работает хорошо:**
- Рестораны с чеками и фото
- Разные люди заказывают разные блюда
- Совместные напитки
- Личные закуски/десерты

⚠️ **Требует внимания:**
- Быстрое питание без чека (как-то фото + ручной ввод)
- Очень размытые фото
- Неидентифицируемые блюда
- Очень большие группы (20+ человек)

## 16. Логика интеллектуального распределения (подробное объяснение)

### Алгоритм распределения расходов:

```
ДЛЯ КАЖДОГО РАСХОДА:
  1. ЕСЛИ есть фото чека И фото стола:
     a) Распознать все позиции на чеке (OCR)
     b) Распознать кто на фото (Face Detection)
     c) Определить какое блюдо перед кем (CV + ML)
     d) Распределить позиции по людям пропорционально потреблению
     e) Для общих блюд (напитки, хлеб) распределить поровну
     
  2. ЕСЛИ есть только банковская транзакция И фото присутствия:
     a) Определить кто присутствовал на фото
     b) Распределить поровну между присутствующими
     
  3. ЕСЛИ нет никаких фото:
     a) Требовать от пользователя ручного ввода или загрузки фото
     
  4) ПРАВИЛО "Личное благо":
     - Если человек заказал ТОЛЬКО для себя → только он платит
     - Если блюдо "опциональное" (дорогое вино) → только заказывающий платит
     - Если блюдо "общее" (хлеб, салат) → распределяется поровну

ВЫХОД: Справедливое распределение с минимальной дополнительной информацией
```

### Примеры логики:

**Пример 1: Общее блюдо**
```
Салат на столе €12
Фото: 3 человека едят из одной тарелки
Результат: каждый платит €12/3 = €4
```

**Пример 2: Индивидуальные блюда**
```
Паста (€18) перед Влад → Влад платит €18
Паста (€16) перед Инна → Инна платит €16
Паста (€14) перед Таня → Таня платит €14
Итого: €48, каждый платит свое
```

**Пример 3: Смешанный сценарий**
```
Основное блюдо (€18) → Влад (€18)
Основное блюдо (€16) → Инна (€16)
Основное блюдо (€14) → Таня (€14)
Салат общий (€12) → €12/3 = €4 каждому
Вода общая (€3) → €3/3 = €1 каждому
Хлеб (€2) → €2/3 = €0.67 каждому

Итого:
- Влад: €18 + €4 + €1 + €0.67 = €23.67
- Инна: €16 + €4 + €1 + €0.67 = €21.67  
- Таня: €14 + €4 + €1 + €0.67 = €19.67
```
